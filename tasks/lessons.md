## boundary — データ型・変換・境界契約
### collisionStart で結果を確定する
- **症状**: x 座標の一発判定でビンを決めると、物理挙動と集計がずれて理論分布に寄らなかった
- **原因**: 連続座標を離散結果へ変換する境界が、衝突ではなく任意のライン通過になっていた
- **対策**: planck.js の bin sensor で着地を確定し、最終ビンの境界契約を物理イベントに合わせる

### Edge レールは貫通しやすい
- **症状**: 斜めのガードレールを線分のまま置くと、速い球がまれに抜けた
- **原因**: ゼロ厚みの Edge は連続衝突判定でも不利で、接触が取りこぼされることがある
- **対策**: 物理レールは回転した厚み付き Box に置き換え、見た目だけ線として描く

## architecture — 設計・責務・config
### Three.js: settled オブジェクトを配列から即座に除去する
- **症状**: サンプル数が多いほど `tosses.forEach` のフレームごとの走査コストが増大し続けた
- **原因**: `settleToss()` でメッシュを dispose・scene から除去していたが、toss オブジェクト自体は `tosses` 配列から削除していなかった。1000サンプルなら 1000 エントリが蓄積したまま毎フレーム走査された
- **対策**: `step()` の `forEach` 直後に逆順ループで `tosses.splice(i, 1)` して settled エントリを除去する。逆順にするのは splice 後のインデックスずれを防ぐため

### Three.js: dispose は一度だけ呼ぶ設計にする
- **症状**: `clearTosses()` がリセット時に、`settleToss()` で既に dispose 済みのメッシュに対して再度 dispose を呼んでいた
- **原因**: dispose 済みかどうかを追跡せず、配列に残っている全エントリに対して無条件に dispose を実行する構造だった。Three.js は冪等処理するためクラッシュしないが、破棄済みリソースを再破棄しようとする構造が壊れていた
- **対策**: settled 済みエントリを配列から即座に除去する（上記）ことで、`clearTosses()` は未 dispose のエントリだけを処理するようになり、二重 dispose が構造的に起きなくなる

### ヒストグラムの乱数は seed 付き生成器を渡す
- **症状**: サンプル生成が reroll ごとにぶれやすく、分布の再現性も落ちた
- **原因**: 乱数ファクトリをそのまま渡していて、実際の生成器を 1 回だけ作る構造になっていなかった
- **対策**: seed から生成した関数を `sample()` に渡し、reroll では seed を進めて再計算する

### main から Galton board を分離する
- **症状**: main.js が分布UIと Galton board の物理・描画・集計を一緒に抱えていた
- **原因**: 初期実装を単一ファイルで育てていて、責務境界が広がっていた
- **対策**: Galton board 固有の処理を `galton-board.js` に移し、main は共通UIと配線に集中させる

### 自動開始は初期化と分ける
- **症状**: ページを開いた時点で Galton board のシミュレーションが勝手に走り始めた
- **原因**: 初期化処理の末尾でボール生成まで実行していた
- **対策**: 初期化は待機状態までに止め、再生成ボタンを押した時だけ走らせる

### 下側敷居は固定高さにしない
- **症状**: 段数を増やすと下側の敷居がペグ領域に食い込み、ボールが詰まった
- **原因**: bin の開始位置と高さを固定余白で決めていて、最下段の位置と干渉した
- **対策**: ペグ最下段の直下に必要最小限だけ敷居を置き、高さを可変にする

## quality — テスト・CI/CD・品質保証
### 検証スクリプトは実装の回転符号と一致させる
- **症状**: 実装は正しいのに、検証スクリプトが前後の回転符号を逆にしていて「漏斗が逆」と誤判定した
- **原因**: 本体の `setRotation()` の符号と、検証側で仮定した符号が一致していなかった
- **対策**: 検証は実装の回転符号をそのまま反映し、左右前後をまとめて判定する

## ui — フロントエンド・デザイン・VRM
### [サブカテゴリ: タイトル]

### 六角形ペグと初期サンプル数を物理に合わせる
- **症状**: 円形ピンの見た目と接触回数の集計が合わず、分布が理論から外れた
- **原因**: 物理 fixture の形状が見た目と一致しておらず、サンプル数も小さすぎた
- **対策**: ピンを六角形 fixture に変え、初期サンプル数を増やして接触統計を先に安定させる

### canvas 固定化で物理再初期化を避ける
- **症状**: 画面サイズ変更のたびに物理シーンが再構築され、分布と形が変わってしまった
- **原因**: canvas サイズを viewport に追従させていたうえ、resize 時に `resetPhysics()` を呼んでいた
- **対策**: canvas の表示サイズを固定し、resize では再初期化しない

### 段数変更と物理サイズの依存を切る
- **症状**: `steps` を変えるたびにボールやペグの見た目サイズまで変わり、詰まり方も不安定になった
- **原因**: ピッチとサイズを `binWidth` に直接ぶら下げていて、段数変更が物理スケール全体を動かしていた
- **対策**: ピッチの基準を固定値にして、段数は配置数だけ変える

### 0.01% の入力はスライダーではなく数値欄にする
- **症状**: レア率のような極小確率をスライダーで扱うと、入力粒度が粗すぎた
- **原因**: `range` の step が大きく、0.01% のような値を直接打てなかった
- **対策**: 確率は `%` 単位の `number` 入力にし、累積到達率グラフとセットで見せる

### canvas 固定後は物理スケールも固定値に戻す
- **症状**: canvas を固定したのに、画面幅由来のピッチが残っていて、要素が大きくなりすぎた
- **原因**: 表示領域の固定と物理スケールの固定が分離できていなかった
- **対策**: 物理ピッチを固定値にし、canvas サイズ変更と物理サイズ変更を切り離す

### 画面幅が大きすぎるときはシェルとピッチを両方下げる
- **症状**: 描画範囲が横に広すぎ、ガードレールもボールも大きく見えた
- **原因**: 固定 canvas に対して board 幅とレールが広すぎた
- **対策**: canvas シェルを小さくし、物理ピッチも小さい固定値に戻す

### ガードレールは見た目と物理厚みを一致させる
- **症状**: 線で描いたガードレールと、実際の厚み付き fixture の隙間が一致しなかった
- **原因**: 描画は中心線だけ、物理は回転矩形で別形状だった
- **対策**: 物理 fixture と同じ厚みの板として描画し、見た目と当たり判定を揃える

### pause ボタンは配線を忘れると無効になる
- **症状**: 一時停止ボタンの見た目はあっても、クリックしてもシミュレーションが止まらなかった
- **原因**: UI 要素を作っただけで、クリックイベントの接続が抜けていた
- **対策**: ボタンの状態切り替えと物理停止を同じイベントで結び、実際に止まる経路を持たせる

### 外側ペグ列でガードレールを補強する
- **症状**: ガードレールを残したままだと、ペグ外縁とレールの見た目のつながりが弱かった
- **原因**: ペグ配置が三角形の内側だけで終わっており、外周の輪郭がレールに届いていなかった
- **対策**: 既存レールは維持しつつ、各段のペグを左右1列ずつ外側に広げて輪郭を揃える

### 空リポジトリでは先に運用ファイルと仕様書を置く
- **症状**: 実装前に置き場所がない状態だった
- **原因**: 初期構成が未作成だった
- **対策**: `tasks/` と `.gitattributes` を先に整え、その上で `README` と詳細仕様を作る

### 3D分布プロトタイプは依存なしで先に画面を置く
- **症状**: 実装の足場がなく、確認できる画面がなかった
- **原因**: 技術選定前に空の状態だった
- **対策**: まずは静的 HTML/CSS/JS で UI とシミュレーション器を置き、後から物理エンジンを差し込む

### 3D scene は結果数を種類ごとに切り替える
- **症状**: 3D のサイコロを 10 面に広げたとき、集計ビンや理論値が 6 面固定のままだった
- **原因**: 見た目の種類と outcome 数を同じ固定値として扱っていた
- **対策**: kind ごとに outcome 数を持たせ、サンプル生成・ビン数・理論分布を同じ値から組み立てる

### 統計検証用の上限は生成速度も同時に調整する
- **症状**: サンプル上限だけ増やすと、再生成が長時間化して検証しづらくなった
- **原因**: サンプル数と spawn 間隔が別々に固定されていた
- **対策**: 上限を上げる時は、生成間隔もサンプル数に応じて短くして実行時間を抑える

### 二項分布は結果だけでなく試行の束として扱う
- **症状**: 二項分布を単発のコイン結果として扱うと、試行回数と分布の関係が見えにくかった
- **原因**: 1 回の結果と 1 試行の結果を同じレイヤーで処理していた
- **対策**: 1 trial = 複数回のコイン投げとして集計し、終わったコインはすぐ消して次の trial に進める

### 超幾何分布は抽選機そのものを分離する
- **症状**: 戻さない抽選をコイン分布の scene に混ぜると、表現がぼやけた
- **原因**: 抽選箱の物理構造と他のコイン系の trial 構造が違うのに、同じ scene に押し込めていた
- **対策**: ビンゴマシンを独立 scene にし、回転・取り出し口・玉の減少を抽選機の責務として切り出す

### ビンゴマシンは trial の取り出し数で打ち切る
- **症状**: 母集団の玉がすべて外へ出るまで待つと、超幾何分布の「戻さない抽選」ではなく「全玉回収」になってしまった
- **原因**: 物理シミュレーションの都合で、抽出終了条件を試行回数ではなく球の排出完了に結びつけていた
- **対策**: 1 trial あたりの取り出し数を上限にして、出口は 1 個ずつ選ぶ。残り玉は次の trial に持ち越さず、trial 終了時にまとめて破棄する

### bingo-machine は Rapier へ責務を寄せる
- **症状**: 自前の簡易物理では球同士・壁・出口の相互作用が不安定で、退出タイミングも信用できなかった
- **原因**: 衝突解決と出口判定をアプリ側で模倣していて、実際の剛体シミュレーションとズレていた
- **対策**: 球・固定壁・キinematic gate・sensor を Rapier に移し、scene は集計と表示だけを担当させる

### MVPはパラメータ操作と理論曲線まで入れて完成とみなす
- **症状**: 画面があるだけで、分布の比較と調整が弱かった
- **原因**: ヒストグラム表示だけではMVP要件が足りなかった
- **対策**: sample数・分布パラメータを操作可能にし、理論曲線を重ねて比較できる状態まで作る

### Galton board は planck.js の実衝突で作る
- **症状**: 釘の見た目はあっても、実際の衝突がないと説得力が弱い
- **原因**: 擬似アニメーションだけでは物理の説得力が出ない
- **対策**: 既存の物理エンジンを導入し、球・釘・仕切り・センサーで実際の衝突と落下を表現する
